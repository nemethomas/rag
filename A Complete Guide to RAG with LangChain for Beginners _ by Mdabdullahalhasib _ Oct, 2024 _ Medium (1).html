A Complete Guide to RAG with LangChain for Beginners

Explore the concepts of RAG (Retrieval Augmented Generation), understand the architecture & implement it with LangChain

Mdabdullahalhasib

·

8 min read

Suppose now you are using a Large Language Model(LLM) which was trained in 2020. If you ask some questions to model that happened in recent days, then the model can’t tell the answer or it will provide the wrong answer. Because the model was not trained with the recent data. So, It has no information about the recent topics. This is one problem. Another problem is that the model is trained on public data. But most of the data in the world are private. So, the model lacks knowledge about private data and it can’t deal well at the time of private data.

When The models lack information, they generate incorrect but believable responses is known as Hallucination. They will provide information confidently and sound accurate but incorrect information due to insufficient knowledge. Due to this issue, Two solutions come into Place such as fine-tuning LLMs & RAG. Today we will explore the RAG in more depth. Let’s start!!!

What is RAG?

RAG is a system that combines the model’s trained knowledge & external data to get appropriate responses to the user queries. Instead of depending on pre-trained knowledge of LLM, the RAG system allows the model to dynamically fetch information from external sources at the time of generating responses. This fetching or retrieval mechanism ensures that the information given to the LLM from external sources is contextually relevant, accurate, and up-to-date. Also, you can provide private data to the LLM and generate appropriate responses.

Through RAG, You get the information of the model’s pre-trained knowledge and additional or domain-specific information using external data sources to generate effective responses. You don’t need to retrain the model repeatedly or fine-tune the model on updated data. It bridges the gap between generative models and external knowledge.

How the RAG system works

First, we collect our external data as Documents, then split the large documents into a number of small chunks. Each chunk can be taken 1000 words or a small number of words. Then convert the chunk’s text into numbers through the Embedding technique. Then Store all the Embeddings in a Vector Database.

Now when User gives their queries to the system. Then the queries are converted into embeddings, extracted relevant context from the Vector database, and passed the context to the prompt template. The user queries also pass to the prompt template. Then the prompt template passes to the LLM and then the model responds based on the prompt.

Here in the model, we provide the user queries and relevant context for queries through LLM. So, LLM models can use their pre-trained knowledge as well as outside context to generate the response. This is the overall procedure of the RAG system. Let’s dive more.

Architecture of RAG

The main components of RAG are Indexing, Retrieval & Generation. In advanced terms, You will get Query Translation, Routing, Query Construction, Advanced Indexing techniques, CRAG, Adaptive RAG, etc. Let’s understand from main components.

Indexing

It’s a pipeline for gathering data from different data sources like websites, pdf, or other documents and indexing those data into a database. It consists of data loading, data splitting, Embedding, & storing data.

LangChain provides a number of functions of Document loaders. You can load PDF, HTML, Website data, API, and others. After loading the data, You have to split the long documents into small chunks. We can’t pass the long documents into the model. In the long chunks, it is hard to search the relevant context against user queries and the model can take a finite number of tokens as input. That’s why we split the text into smaller parts that are useful for both indexing data and passing the data into the model. After splitting documents, we have to apply the Embedding techniques to convert the text chunks into numerical format as vectors. The embedded vector will be stored in a vector database and indexing each split so that they can be searched over.

Retrieval & Generation

It has two parts such as Retrieve & Generation. When a user provides the queries to the system. Then the queries convert into numbers through embeddings. After the user embeddings search the most relevant chunks from the vector store with the help of the retrieval techniques such as cosine similarity or others. Then extract the most relevant context from the vector store and pass it to the prompt. user queries also go to the prompt without any modification. Then prompt goes to the LLM model. Then the model an answer based on the prompt that includes user queries and retrieved data.

Advanced Techniques in RAG

To solve various issues, the innovation of RAG is increasing day by day such as Query Translation, Routing, Query Construction, advanced indexing techniques, CRAG, Adaptive RAG, and so on.

Query Translation is for translating and refining queries in different ways such as Multi-Query Techniques, RAG Fusion, Decomposition, Step Back, HyDE approaches, and so on. Routing Queries are used to determine the best modern for processing the request using metadata, query type, or historical data. Query Construction involves keyword extraction, synonym generation, and contextual understanding to formulate queries that can accurately reflect the user’s intent. Advanced Indexing techniques like RAPTOR and Colbert are used to rapidly retrieve relevant documents from the Vector database. CRAG(Conditional RAG) is used to introduce conditional logic into the RAG system to adjust retrieval strategies based on the specific conditions of a query. Adaptive RAG extends the flexibility of traditional RAG systems by incorporating feedback loops that allow the system to learn from previous interactions. These are some advanced techniques of RAG to create more effective RAG applications.

Implementation Of RAG

The techniques to perform an RAG application are Document Loaders, Text Splitting, Embedding, Vector Stores, Retrievals, ChatModels, Key-Value Stores, and others. The step-by-step guide to implementing the RAG application is:

  • Loading the Chat Model
  • Loading all the Documents
  • Splitting the documents into multiple small chunks
  • Performing Embeddings & storing those in the Vector Database
  • Retrieve Contexts from the Database based on User Question
  • User Questions & Retrieved contexts are sent to the prompt templates
  • Sending prompt template to LLM and Generate response.

These are the step by step methodologies to implement the RAG system. Let’s get started with importing libraries of LangChain & Loading the Gemini model as LLM.

# Import necessary libraries
from langchain_core.output_parsers import StrOutputParser
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
import os
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_google_genai import GoogleGenerativeAIEmbeddings
from dotenv import load_dotenv

load_dotenv()
api_key = os.getenv("GOOGLE_GEN_API")

llm = ChatGoogleGenerativeAI(model="gemini-1.5-pro", temperature=0.3, api_key=api_key)
llm.invoke("what is langchain")

Load Documents: We have used a Python book to load by PyPDFLoader. You can load other Documents as well through LangChain’s Document Loader.

def load_documents(file_path):
    loader = PyPDFLoader(file_path)
    documents = loader.load()
    return documents

pdf_path = 'data/python book.pdf'

documents = load_documents(pdf_path)

Split Documents: It is used to split large documents into smaller chunks. A popular recursive Character Text Splitter is used here where each chunk will contain 10000 words and 100 words will overlap between two chunks. You will get more methods of Splitting in Langchain.

# Split documents into smaller chunks
def split_documents(documents):
    text_splitter = RecursiveCharacterTextSplitter(chunk_size=10000, chunk_overlap=100)
    split_docs = text_splitter.split_documents(documents)
    print(len(split_docs))
    return split_docs

split_docs = split_documents(documents)

we get the smaller chunks. Now we have to perform the embedding to convert the chunk’s text into numerical format and then store the embeddings into the Vector database. Here, we have used Google Generative AI Embeddings for performing embeddings. You can also use OpenAI API, Huggingface, or other providers to perform embeddings. Then FAISS is used for storing and indexing the embeddings that can be stored locally. Chomadb, pinecone, and others are some other options for vector databases. Let’s see the code.

#pip install faiss-cpu / faiss-gpu
from langchain.vectorstores import FAISS

def get_vector_store(text_chunks):
    texts = [chunk.page_content for chunk in text_chunks]  # Extract text content from the chunks
    embeddings = GoogleGenerativeAIEmbeddings(model = "models/embedding-001",google_api_key=api_key)
    vector_store = FAISS.from_texts(texts, embedding=embeddings)
    vector_store.save_local("faiss_index")

get_vector_store(split_docs)

Now we have to retrieve the contexts from the database based on the User Question. First, it is needed to load the database and extract similar contexts for user questions through cosine similarity search or others.

def retreive_context(user_question):
    embeddings = GoogleGenerativeAIEmbeddings(model = "models/embedding-001",google_api_key=api_key)
    new_db = FAISS.load_local("faiss_index", embeddings,allow_dangerous_deserialization=True)
    docs = new_db.similarity_search(user_question)
    print(docs)
    return docs

retreive_context("Which topics are covered in the book")

We defined the prompt template where context is retrieved-context based on user queries and questions are the provided user queries. Then prompt is passed to llm and generates the response. Here in the prompt, we provide the user question and updated/private information related to the question and generate an appropriate response. That’s the RAG.

from langchain_core.runnables import RunnableLambda, RunnablePassthrough
from langchain_core.prompts import ChatPromptTemplate

messages = [
    ("system", "You are an assistant for question-answering tasks."),
    ("human", "Use the following pieces of retrieved context to answer the question. \n\
    If you don't know the answer, just say that you don't know. Use three sentences maximum and keep the answer concise. \n\
    Question: {question} \n\
    Context: {context} \n \
    Answer:")
]

prompt = ChatPromptTemplate.from_messages(messages)



def format_docs(docs):
    return docs[0].page_content


rag_chain = (
    {"context": RunnableLambda(lambda x: retreive_context(x)) | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

rag_chain.invoke("Which topics are covered in the book?")

If you are interested in running the RAG system based on OPENAI API and Chromadb database. Then You can run the following code to understand the RAG better.

import bs4
from langchain import hub
from langchain_chroma import Chroma
from langchain_community.document_loaders import WebBaseLoader

from langchain_core.runnables import RunnablePassthrough
from langchain_openai import OpenAIEmbeddings
from langchain_text_splitters import RecursiveCharacterTextSplitter

# Load, chunk and index the contents of the blog.
loader = WebBaseLoader(
    web_paths=("https://lilianweng.github.io/posts/2023-06-23-agent/",),
    bs_kwargs=dict(
        parse_only=bs4.SoupStrainer(
            class_=("post-content", "post-title", "post-header")
        )
    ),
)
docs = loader.load()

text_splitter = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200)
splits = text_splitter.split_documents(docs)
vectorstore = Chroma.from_documents(documents=splits, embedding=OpenAIEmbeddings())

# Retrieve and generate using the relevant snippets of the blog.
retriever = vectorstore.as_retriever()
prompt = hub.pull("rlm/rag-prompt")


def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)


rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

rag_chain.invoke("What is Task Decomposition?")



# cleanup
vectorstore.delete_collection()
